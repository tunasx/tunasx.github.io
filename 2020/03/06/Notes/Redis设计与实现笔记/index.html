REDIS设计与实现笔记

Mr. Tunas

notes

参考《redis设计与实现第二版》

数据结构和对象

SDS简单动态字符串

  1. 二进制安全
    除了可以保存文本文件外，还可以保存图片、音频、视频、压缩文件等二进制数据
  2. 遵循C字符串空字符结尾， 可直接重用部分C字符串库函数
  3. 长度获取 O(1)
  4. 杜绝缓存区内存溢出
  5. 减少字符串修改时的内存重分配次数
    通过SDS的未使用空间，实现空间的预分配和惰性空间释放两种优化策略
    1). 预分配:
    if sizeof(newStr) < 1MB free = sizeof(newStr)
    if sizeof(newStr) >= 1MB free = 1MB
    2). 惰性空间释放
    当需要缩短SDS的字符串长度是， 并不立即进行内存重分配来回收多余的内存， 而是使用free记录，以备将来使用

链表

实现： 正常的双端链表

特点

  1. 双端, 获取某个节点的前后节点时间复杂度O(1)
  2. 无环, 以NULL为终点
  3. 带表头和表尾指针，获取表头和表尾均为O(1)
  4. 带链表计数器，获取链表长度O(1)
  5. 多态

字典

哈希表

Reids使用的hash计算算法：MurmurHash2

rehash
当哈希表的负载因子到达一定程度时需要进行扩展和收缩
rehash步骤

  1. 为ht[1]分配空间，大小根据要执行的操作来确定
    • 如果是扩展操作，ht[1]的大小为第一个大于等于ht[0].used*2的2^n
    • 如果是收缩操作，ht[1]的大小为第一个大于等于ht[0].used的2^n
  2. 将h[0]的所有键值对rehash到h[1]上面: rehash就是重新计算键的哈希值和索引值，然后将键值对放置在ht[1]哈希表的指定位置
  3. 当h[0]中的所有键值对都迁移到ht[1]后，释放h[0]，将h[1]设置为ht[0]，并在h[1]新创建一个空的哈希表，为下一次rehash做准备

哈希表的扩展和收缩时机
负载因子计算：

下面任意条件满足会进行扩展：

  1. 服务器没有在执行BGSAVE或者BGREWRITEAOF，并且负载因子>=1
  2. 服务器正在执行BGSAVE或者BGREWRITEAOF，并且负载因子>=5
    当哈希表的负载因子<0.1时，程序自动开始对哈希表进行收缩操作

渐进式rehash
rehash不是一次性、集中式完成的，而是分多次、渐进式地完成的
在每次操作字典时顺带对当前的rehashindex索引进行rehash（每次 rehashindex 增一）， 避免了集中式rehash带来的庞大计算量

跳跃表

跳跃表支持平均O(logN)、最坏O(N)复杂度的节点查找
跳跃表在Redis中仅两个地方使用： 1. 实现有序集合的键 2. 集群节点中作为内部数据结构

压缩链表的结点构成

每个亚索列表节点可以保存一个字节或者一个整数值，其中，字节数组可以是一下几种:

1 <= 63(2^6 - 1)字节的字节数组
2 <= 16383(2^14 - 1)字节的字节数组
3 <= 4294967295(2^32 - 1)字节的字节数组

整数为如下几种:

1 4位长，介于0～12之间的无符号整数
2 1字节长的有符号整数
3 3字节长的有符号整数
4 int16_t类型整数
5 int32_t类型整数
6 int64_t类型整数

previous_entry_length
标示前一个节点的长度，单位为字节，通过指针运算可以计算前一个节点的起始位置

  1. 前一个节点长度小于254时， 占用1字节
  2. 前一个节点长度大于254时， 占用5字节

encoding
记录content属性所保存数据的类型以及长度

  • 一字节、两字节或者五字节长， 值的最高位为00、01或者10表示content属性是字节数组 数组长度由编码去除最高两位之后的表示
  • 一字节长，值的开头是11，表示content属性是整数 整数类型和长度由编码去除最高两位之后的表示

RDB

SAVE 和 BGSAVE

SAVE会阻塞进程
BGSAVE 不会阻塞主进程，通过fork子进程来进行rdb文件的生成

服务器启动的时候加载RDB文件

AOF的优先级高于RDB
AOF关闭状态下才使用RDB

BGREWRITEAOF和BGSAVE不能同时执行

dirty计数器和lastsave属性

AOF实现持久化

命令追加 文件写入 文件同步

flushAppendOnlyFile
策略
always 将aof_buf中的内容写入同步到aof文件
everysec 写入aof文件，如果距离上次操作超过1秒便进行同步，并且同步操作由另一个线程负责
no 只写入aof文件不进行同步，同步操作由系统决定

Redis的主从复制的实现

完整重同步
部分重同步（复制积压缓存区）

  1. 从服务器接收到同步命令
  2. 从服务器向主服务器开始套接字连接
  3. 连接创建成功，检查连接的网络情况发送PING
  4. 收到主服务器的PONG相应，开始复制
  5. 有需要时发送AUTH身份验证
  6. 发送端口信息
  7. 同步
  8. 命令传播

  5. 创建向主服务器的网络连接
